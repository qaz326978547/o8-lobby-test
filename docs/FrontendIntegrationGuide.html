整體流程概觀

跟「UGS 自己的大廳頁面」不同，前後端分離的合作方不會經過 UGS 的 302 轉址流程。流程改成：

  1. 合作方後端自行向 Brand API 取得 player 的 token（即 authtoken）。
  2. 合作方後端把 token 帶到自己的前端網址，例如 https://your-frontend.example.com/lobby?token=xxx。
  3. 前端（瀏覽器）從網址上的 query string 取出 token，之後呼叫 UGS 的大廳 API（列表 / 搜尋 / 啟動遊戲）時，每一支 API 都要帶上這個 token（query string）。
瀏覽器網址：https://your-frontend.example.com/lobby?token=eyJxxx...

前端拿到 token 後呼叫：
GET {UgsFrontendWebsiteUrl}/api/lobby/{lobbyPath}?token=eyJxxx...
GET {UgsFrontendWebsiteUrl}/api/lobby/{lobbyPath}/search?token=eyJxxx...&keyword=SG&pageSize=10
GET {UgsFrontendWebsiteUrl}/gamelauncher?token=eyJxxx...&gpcode=SG&gcode=SGCaishen&betlimitid=1&lang=zh-CN
重點：「自己的前端網址」是合作方自行決定、自行部署的，UGS 不需要、也不會登錄這個網址。UGS 只負責「給 token」與「吃 token 回資料」。

Step 1：取得 token

由合作方後端（不是瀏覽器）執行，流程與既有 Brand 整合方式相同：

步驟說明
1. 取得 Brand Token POST {BrandApiUrl}/connect/token（grant_type=client_credentials），取得 access_token
2. AuthorizePlayer 用 access_token 呼叫 AuthorizePlayer API，取得該玩家的 authtoken

這個 authtoken 就是後續所有大廳 API 要帶的 token 參數。

Step 2：組出自己的前端網址

合作方後端把 authtoken 帶到自己的前端入口：

https://your-frontend.example.com/{lobbyPath}?token={authtoken}

{lobbyPath} 是大廳代碼（例如 desktop、o8），由 UGS 提供／約定。

API：大廳列表 既有

GET {UgsFrontendWebsiteUrl}/api/lobby/{lobbyPath}?token={token}

回傳整個大廳資料：Banner 輪播、分類頁籤、各區塊（熱門遊戲 / 新品推薦）與遊戲卡片清單等。前端依回傳的 groups[].isvisible / order 渲染畫面。

API：搜尋（分頁 / 無限滾動） 已實作

GET {UgsFrontendWebsiteUrl}/api/lobby/{lobbyPath}/search?token={token}&keyword=...&pageSize=...&gamesOffset=...&providersOffset=...&gameTypesOffset=...

一個 keyword 同時搜尋「遊戲」「遊戲商」「遊戲類型」三種結果，各自獨立分頁，前端可分別對三段做無限滾動。

Request 參數

欄位位置型別必填說明
lobbyPathroutestring✓大廳路徑
tokenquerystring✓玩家授權 token
keywordquerystring關鍵字（空字串 → 三段皆回傳空結果）
pageSizequeryint各段每頁筆數，預設 20，上限 100
gamesOffsetqueryintgames 段起始位置，預設 0
providersOffsetqueryintproviders 段起始位置，預設 0
gameTypesOffsetqueryintgameTypes 段起始位置，預設 0

Response 結構

每一段（games / providers / gameTypes）皆為相同的分頁結構：

欄位型別說明
items陣列本頁資料
offsetint本頁起始位置（即帶入的 xxxOffset）
totalCountint符合條件的總筆數（分頁前）
hasMorebool是否還有下一頁
無限滾動規則：當某一段 hasMore = true 時，下一頁請求把該段的 xxxOffset = offset + items.length，其餘 offset 維持原值、keyword 不變即可。三段各自獨立，互不影響。

空結果情境：keyword 為空、或 token/lobbyPath 無效，皆回傳 HTTP 200 與「三段皆空」（items=[]、totalCount=0、hasMore=false），不會丟例外或錯誤碼。前端只需處理「空陣列」畫面，不用特別 catch 錯誤。

API：啟動遊戲 既有

GET {UgsFrontendWebsiteUrl}/gamelauncher?token={token}&gpcode={providercode}&gcode={code}&betlimitid=...&lang=...

使用者點擊遊戲卡片時，帶卡片上的 providercode（gpcode）與 code（gcode）呼叫此 API，取得遊戲 URL 後以 iframe 載入或重導；若該遊戲商不支援 iframe（回傳會標示），改用整頁重導。

CORS 注意事項 後端待辦

因為前端網址跟 UGS API 不同網域，瀏覽器呼叫 API #1/#2/#3 屬於跨網域請求，需要 UGS 後端開啟 CORS 並把合作方的網域加入允許清單。

狀態：此項目目前尚未設定，已記錄在規格文件的待辦中。串接前請與後端團隊確認你的前端網域是否已加入 CORS 允許清單，否則瀏覽器會擋下請求（出現 CORS error，而不是 API 回傳錯誤）。

Token 限制

  • token 即 Brand AuthorizePlayer 回傳的 authtoken，單次/短效。
  • 過期後的刷新機制目前未規劃：若 API 回傳 401（或其他授權失敗），代表 token 已失效。
  • 處理方式：前端導回自己的登入頁，由合作方後端重新走 Step 1取得新的 authtoken，再產生新的 ?token= 網址。
  • 同一個 token 可重複用在多支 API（列表 / 搜尋 / 啟動遊戲），不需每支 API 換 token。

錯誤處理

情境API 行為前端建議
搜尋 keyword 為空HTTP 200，三段皆空結果當作「無結果」畫面顯示，不需特別處理
token 無效 / 過期列表 / 搜尋：HTTP 200 + 空結果或無資料；啟動遊戲：可能 401偵測到資料異常或 401 時，導回登入頁重新取得 token
跨網域被擋瀏覽器顯示 CORS error，無法取得 response確認網域已加入後端 CORS 允許清單
